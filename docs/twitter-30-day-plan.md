# QuietLog — 30 Day Twitter / X Posting Plan

## Goal

Build early trust and interest before launch without posting every day.

QuietLog should not feel like a hype product.
It should feel like a calm, reliable utility being built carefully.

## Posting Cadence

Recommended:

- 3 posts per week
- around 12 posts in 30 days
- optional extra posts only for real milestones

Suggested days:

- Monday: philosophy / product positioning
- Wednesday or Thursday: development progress
- Saturday or Sunday: reliability / behind-the-scenes / screenshot

## Tone

Posts should feel:

- honest
- practical
- calm
- transparent
- low-hype

Avoid:

- engagement bait
- fake urgency
- overpromising
- daily spam
- pretending unfinished features are finished

# Week 1 — Identity And Promise

## Post 1 — Intro

Building QuietLog — a private offline planner for real life and gaming.

No ads.
No cloud.
No account.
No AI.
No subscription.

Just user-created reminders, shopping lists, and game events.

## Post 2 — Problem Statement

Most apps use notifications to pull you back in.

QuietLog is built for the opposite:
only remind you about things you personally created.

Quiet by default.
Useful when needed.

## Post 3 — Trust Positioning

QuietLog is not trying to become another productivity app that wants your attention all day.

The goal is simple:

A small offline planner that respects your time, privacy, and notifications.

# Week 2 — Development Foundation

## Post 4 — Development Start

QuietLog development focus right now:

- Android foundation
- clean navigation
- local storage
- reminder system
- notification reliability

No feature bloat.
No cloud backend.
No account system.

## Post 5 — Offline-First Explanation

Why offline-first?

Because a personal reminder app should not need an account or server to remember your tasks.

QuietLog data stays on your device.
That keeps the app simpler, cheaper, and more private.

# Week 3 — Reminder Reliability

## Post 7 — Reminder Trust

For a reminder app, trust matters more than feature count.

QuietLog will not be pushed publicly until reminders are tested for:

- app restart
- snooze
- duplicate prevention
- reboot behavior
- real device reliability

## Post 8 — First Reminder Flow

Current milestone:

Create reminder
→ save locally
→ show in list
→ trigger notification
→ snooze or mark done

One complete reliable flow is more important than ten unfinished features.

## Post 9 — Pixel Testing Teaser

Real device testing matters for reminder apps.

Emulators are useful, but Android notification behavior needs real hardware testing too.

QuietLog will be tested on a Pixel device before public release.

# Week 4 — Practical Features

## Post 10 — Shopping Feature

Small feature idea in QuietLog:

If you are shopping and stop checking items for a few minutes while items are still open, QuietLog can remind you:

“Still items left on your list.”

No GPS.
No store tracking.
Just simple local logic.

## Post 11 — Game Events

QuietLog will also support manual game event reminders.

Example:

Boss spawns at 20:00.
Notify me 10 minutes before.
Open the linked game from the reminder.

No game API needed.
You control the reminder.

## Post 12 — Call Reminder

Call reminders in QuietLog are designed for comfort, not automation.

You can save a phone number with a reminder.
Tap Call.
Android dialer opens with the number filled in.

QuietLog never calls automatically.
The user stays in control.

Feature rule for QuietLog:

If a feature adds 5% value but 30% complexity, it stays out.

The app should remain small, reliable, and easy to trust.

## Optional — Pricing Philosophy

QuietLog is planned as a low-priced one-time purchase.

No ads.
No subscription.
No account.
No cloud backend.
