# QuietLog — Release Philosophy

## Goal

QuietLog releases should prioritize:

- stability
- reliability
- trust
- clear communication

not feature spam.

---

## Release Style

Prefer:

- smaller stable releases
- focused improvements
- clear patch notes
- predictable iteration

Avoid:

- giant unstable updates
- random feature dumps
- unnecessary redesigns

---

## Versioning Philosophy

Suggested format:

```text
major.minor.patch
```

Examples:

```text
1.0.0
1.1.0
1.1.1
```

---

## Patch Version

Patch releases should contain:

- bug fixes
- small improvements
- reliability improvements

Example:

```text
1.0.1
```

---

## Minor Version

Minor releases should contain:

- meaningful features
- workflow improvements
- UI improvements

Example:

```text
1.1.0
```

---

## Major Version

Major versions should only happen when:

- architecture changes significantly
- app direction changes significantly
- major systems are introduced

Avoid unnecessary major bumps.

---

## Main Rule

A stable small update is better than a large unstable one.
